Today, I'll create a hash verification signature file for clients running download_firmware.py.

The Dev Server workflow will publish three files: firmware.bin, signature.sig, and hash_result.md. Clients should download the firmware.bin, signature.sig, and hash_result.md files by running download_firmware.py. This file includes logger.py to log activity and verify hashes so the hash server can compare the downloaded hashes. So in this situation, a hacker could insert it into the main server by replacing the contents of firmware.bin, signature.sig and hash_result.md which have been modified by the hacker, so with the initial step from the server hash to the local hash, the final step is to monitor the contents of signature.sig because the mechanism of the key pair is 1 private will produce 1 public key, so indirectly we use the power of the public key to be able to recognize this private key whether it is from the developer or from the hacker. This is the final step of the OTA Server automation on the Client side, the Client runs download_firmware.py which contains 3 files from the firmware.bin file, signature.sig and hash_result.md. This will be reviewed from logger.py to monitor the activity of download_firmware.py then it will execute verify_hash.py to compare the contents of the server hash with the contents of the local hash and the final step is to execute verify_signature.py to recognize whether these 3 files are valid from the developer or the hacker via public_key.pem, the results will be valid or invalid
