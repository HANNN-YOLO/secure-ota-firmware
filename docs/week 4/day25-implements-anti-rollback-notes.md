Today, I'll be building the logic for how anti-rollback works on the client.

Basically, there are only three permanent files in the client: the public key to identify whether the file is from a developer or a hacker; the installed_version.json file, which will be compared if there's a semantic version update; and the code that automatically downloads and installs it: download_firmware.py, which contains logger.py to monitor activity; verify_hash to compare the local hash with the server hash; verify_signature_firmware.py to determine whether the file is from a developer or a hacker; anti_rollback.py to compare the current semantic version number with the next semantic version number; verify_signature_version to determine whether the file is from a developer or a hacker; and finally, update_installed_version.py to update the contents of installed_version.js if the download is successful.

The Dev Server workflow will publish five files: version.json, version.sig, firmware.bin, firmware.sig, and firmware_hash.md.
First, the client will download the version.json file with the version.sig file. Then, automated by verify_signature_version.py, the version.sig file will be checked against the client's public key. If successful, the next step will run anti_rollback.py to check whether the downloaded file exceeds the existing version. If the version suddenly drops after downloading, anti_rollback.py will stop downloading files such as firmware.bin, firmware_hash.md, and firmware.sig. If the version is higher than the installed version, it will download the firmware.bin, firmware_hash.md, and firmware.sig files. The client must download the firmware.bin, firmware.sig, and firmware_hash.md files by running download_firmware.py. This file includes logger.py to log activity and verify the hash so that the hash server can compare the downloaded hash. So in this situation, hackers can enter it into the main server by replacing the contents of firmware.bin, firmwaresig, and firmware_hash.md that have been modified by the hacker, so the initial step is from the server hash to the local hash, the last step is to monitor the contents of signature.sig because the key pair mechanism is 1 private key will produce 1 public key, so indirectly we use the strength of the public key to be able to recognize this private key whether it comes from the developer or from the hacker. This is the last step of the OTA Server automation on the Client side. The client runs download_firmware.py which contains 5 files from the firmware.bin, firmware.sig, and firmware_hash.md files. This file will be reviewed by logger.py to monitor the download_firmware.py activity, then it will run verify_hash.py to compare the contents of the server hash with the contents of the local hash, and the last step is to run verify_signature.py to recognize whether these three files are valid from the developer or the hacker via public_key.pem. The result will be valid or invalid. If valid, it will run the final step of the download_firmware.py automation, namely update_installed_version.py, to update the contents of installed_version.py to retrieve the latest semantic version number.
